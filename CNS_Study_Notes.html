Client-Server & P2P Paradigm
TCP / UDP
▾
Client-Server: A model where the Client requests a service and the Server provides it. Server is always ON with a fixed IP. Multiple clients can connect to one server.

P2P (Peer-to-Peer): Every device acts as BOTH client AND server simultaneously. No dedicated server needed — peers share resources directly.
Client-Server Flow
DIAGRAM CLIENT ──── REQUEST ───────► SERVER CLIENT ◄─────── RESPONSE ──── SERVER Server: Always ON, Fixed IP, Waits for requests Client: Initiates connection, Variable IP
P2P Model
[Peer1] ◄──────────────► [Peer2] ▲ ▲ │ │ ▼ ▼ [Peer4] ◄──────────────► [Peer3] Everyone is equal — Everyone shares! Examples: BitTorrent, Bitcoin, Old Skype
TCP 3-Way Handshake

✅ CONNECTION ESTABLISHED

TCP = Phone call | UDP = Postal letter. TCP detects collisions; UDP just sends!

FTP & TFTP — File Transfer Protocols
⭐ IMPORTANT
▾
FTP (File Transfer Protocol): Uses TWO TCP connections — Port 21 (Control) and Port 20 (Data). Has authentication and directory listing.

TFTP (Trivial FTP): Simplified FTP using UDP Port 69. No authentication, no directory listing. Transfers in 512-byte blocks.
FTP Two-Connection Model
FTP DIAGRAM CLIENT ═══ Control Connection (Port 21) ═══ SERVER ← Commands: USER, PASS, RETR, STOR (stays open whole session) CLIENT ─── Data Connection (Port 20) ─── SERVER ← Actual file content (opens & closes per transfer)
TFTP Block Transfer
CLIENT ── RRQ (filename) ──────────► SERVER CLIENT ◄── DATA Block #1 (512 bytes) ── SERVER CLIENT ── ACK #1 ──────────────────► SERVER CLIENT ◄── DATA Block #2 (512 bytes) ── SERVER CLIENT ── ACK #2 ──────────────────► SERVER CLIENT ◄── DATA Block #N (<512 bytes)── SERVER ← Last block! CLIENT ── ACK #N ──────────────────► SERVER ← Transfer complete ✅
FTP vs TFTP Comparison
FeatureFTPTFTP
Port20, 2169
TransportTCPUDP

FTP: 20=Data, 21=Control. TFTP: T=Trivial=Simple=UDP. Last block <512 bytes = end of transfer!

SMTP — Simple Mail Transfer Protocol
⭐ IMPORTANT
▾
SMTP is used to SEND emails between servers. Uses TCP Port 25. Uses store-and-forward mechanism. Cannot receive emails — needs POP3/IMAP for that.
SMTP Communication Flow
1
SERVER → 220 Service Ready (Server greets)
2
CLIENT → HELO client.com (Introduce itself)
3
CLIENT → MAIL FROM: <alice@a.com> → Server: 250 OK
4
CLIENT → RCPT TO: <bob@b.com> → Server: 250 OK
5
CLIENT → DATA → Server: 354 Start sending
6
CLIENT → [email body] → "." → Server: 250 Message accepted
7
CLIENT → QUIT → Server: 221 Bye ✅
Complete Email Flow
ALICE ──SMTP(25)──► [Alice's Server] ──SMTP(25)──► [Bob's Server] ──POP3/IMAP──► BOB SMTP = Sending (Push) ✈️ POP3/IMAP = Receiving (Pull) 📥
SMTP Response Codes
CodeMeaning

250OK — command accepted
354Start mail input
550User not found
421Service unavailable
SMTP = Sends Mail To People. SMTP can only SEND — for receiving use POP3 (Port 110) or IMAP (Port 143)!

MAC Sub Layers — DCF & PCF
⭐⭐ VERY IMPORTANT
▾
DCF (Distributed Coordination Function): PRIMARY, MANDATORY. Uses CSMA/CA. All devices compete fairly.
PCF (Point Coordination Function): OPTIONAL. AP polls devices one by one. No collisions.
Why CSMA/CA (not CSMA/CD)?
Wired (CSMA/CD): Can detect collision by voltage change ✅ Wireless (CSMA/CA): Cannot detect own collision ❌ Device can't hear itself while transmitting! → Must AVOID collision before sending
DCF Working Steps
1
SENDER senses channel → Is it free for DIFS time?
2
If FREE: Wait random Backoff time → sense again
3
If still free → Send DATA frame
4
Wait for ACK from receiver
5
ACK received → SUCCESS ✅ | No ACK → Retransmit ❌
Timing Intervals
TIMELINE ───[DIFS]──[Backoff]──[DATA]──[SIFS]──[ACK]────► (wait) (random) (send) (short) (confirm) DIFS = DCF InterFrame Space (longest wait — before data) SIFS = Short InterFrame Space (shortest — before ACK) Backoff = Random timer to avoid simultaneous transmission
DCF vs PCF
FeatureDCFPCF
TypeContention-basedContention-free
Mandatory?Yes (Mandatory)No (Optional)
ControllerDistributed (all devices)Centralized (AP)
MethodCSMA/CAPolling
CollisionPossible (but avoided)No collision
ModeInfrastructure + Ad-hocInfrastructure only
DCF = Democracy (everyone competes). PCF = Dictatorship (AP decides who talks). DCF is MANDATORY, PCF is OPTIONAL!

Hidden & Exposed Station Problem
⭐⭐ VERY IMPORTANT
▾
1. Hidden Station Problem
Hidden Station: Two stations are out of range of each other but both communicate with the same AP → causes collision at AP without either knowing!
A
Station A
AP
Access Point
B
Station B
A cannot hear B ❌ | B cannot hear A ❌ | Both can hear AP ✅ → Both transmit simultaneously → COLLISION at AP!
RTS/CTS Solution
1
A → RTS (Request To Send) to AP — "I want to send data"
2
AP → CTS (Clear To Send) broadcast — heard by ALL including B
3
B hears CTS → "Someone is sending, I must WAIT ⏳"
4
A → DATA → AP → ACK → No collision ✅
2. Exposed Station Problem
Exposed Station: A station unnecessarily stops transmitting because it hears a nearby transmission — even though its own transmission would NOT cause any collision!
EXPOSED STATION [A] ◄──── [B] ──────── [C] ────► [D] B can hear C (C is sending to D) B wants to send to A — safe! A cannot hear C But B senses C's transmission → B WAITS unnecessarily ⏳ WASTED bandwidth! B→A would NOT interfere with C→D
Comparison
FeatureHidden StationExposed Station
ProblemCauses collision at APUnnecessary waiting
Channel appearsFree (but it's not!)Busy (but safe to send!)
ResultCollision occursBandwidth wasted
SolutionRTS/CTSNo perfect solution
SeverityMore seriousLess serious
Hidden = Can't SEE each other → COLLISION. Exposed = CAN SEE → unnecessary WAITING. Hidden is more dangerous!
